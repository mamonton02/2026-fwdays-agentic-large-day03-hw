# Spec: Improve Feedback: Display Error Message for Invalid Hexadecimal Color Codes #9527

## Goal
Improve user feedback in the custom color hex input by showing clear validation errors when the entered value is not a valid hexadecimal color format.

## Context
Issue #9527 reports that invalid or malformed hex values in the color input are currently ignored without any visual or textual feedback. As a result, users may not understand why the selected tool color does not change.

Reported invalid examples include:
- `123456789` (more than 8 characters)
- `1` (1 character)
- `12` (2 characters)
- `12345` (5 characters)
- `1234567` (7 characters)
- `zzzzzz` (invalid non-hex characters)
- `blue` (non-hex text)

Expected behavior from the issue is to display a visible error message such as:
- Hex code must be 3, 4, 6, or 8 characters (excluding `#`)
- Invalid characters in hex code
- Not a valid hex color

## Component / Module
 Hex validation and normalization are centralized in normalizeInputColor in packages/common/src/colors.ts (built on tinycolor). The rest of the picker (sliders, palette, etc.) lives alongside it under packages/excalidraw/components/ColorPicker/ (e.g. Picker.tsx, ColorPicker.tsx).

## Non-goals
- Redesigning the full color picker UX beyond validation feedback.
- Changing how valid hex colors are parsed or applied.
- Adding support for non-hex color formats (for example, named colors like `blue`, `rgb()`, or `hsl()`).
- Modifying unrelated style, theme, or palette behavior.

## Acceptance Criteria
- Invalid hex input triggers clear and visible feedback (inline text, tooltip, or equivalent error UI).
- Validation enforces allowed hex lengths of 3, 4, 6, or 8 characters (excluding optional leading `#`).
- Validation rejects non-hexadecimal characters.
- The color is not silently ignored: users are explicitly informed when input is invalid.
- Valid hex values continue to apply color correctly with no regression to current behavior.
